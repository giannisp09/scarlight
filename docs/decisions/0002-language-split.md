# ADR 0002 — Language split: Python control plane, Rust trust path, TypeScript UI, WASM skills

**Status:** Accepted
**Date:** 2026-05-13

## Context

Scarlight has heterogeneous components with different requirements:

- **Skills, agent loop, eval runners** need to integrate with the offensive-security ecosystem (pwntools, scapy, impacket, mitmproxy, requests, beautifulsoup, pwndbg, angr). Most of this is Python.
- **Aegis gateway, credential vault, scope enforcer** are on the security-critical path. Garbage collection pauses and memory unsafety are not acceptable here.
- **UI** (when it exists) is a web UI for ops visibility, replay, scoreboard.
- **Skill-tier sandbox** (untrusted, agent-generated skills) needs capability-restricted execution. WASM is the only mature OSS option.

## Options considered

| Component | Option A | Option B | Option C |
|-----------|----------|----------|----------|
| Control plane | Python | Go | Rust |
| Trust path | Rust | Go | Python |
| UI | TypeScript | Go templates | Rust + WASM frontend |
| Skill sandbox | WASM | Firecracker only | gVisor |

## Decision

- **Python** for: Hydra, Forge (control logic), Codex (skill runtime for ecosystem skills), Mnemos (high-level memory APIs), Phoenix orchestrator, Crucible runner.
- **Rust** for: Aegis (gateway, vault, scope enforcer, capability tokens), Lighthouse audit-journal writer (signing path), Mnemos low-level storage if profiling demands it.
- **TypeScript** for: web UI (Lighthouse replay, scoreboard dashboards).
- **WASM** for: skill-tier sandbox; first execution tier for new / agent-generated skills before MicroVM promotion.

## Rationale

**Python for control & skills.**

- Offensive-security ecosystem is Python-native. pwntools, impacket, scapy, mitmproxy, requests, sqlmap-as-library — switching languages loses 5–10 years of community work.
- Skills are short, often interactive, often glue. Python's expressiveness pays off.
- Frontier-model SDKs (anthropic, openai, google-genai, vLLM) are Python-first.
- CAI, HackSynth, PentestGPT, HackingBuddyGPT — every credible prior art is Python.
- Cost: GIL + GC. Mitigated by `asyncio` + multi-process workers; Rust takes the hot path where it matters.

**Rust for the trust path.**

- Aegis enforces scope, signs audit entries, holds credentials. A memory-safety bug here is catastrophic — exactly the OpenClaw CVE-2026-25253 class.
- Rust's `forbid(unsafe_code)` + capability-style API design produces the strongest static guarantees available without going to formal methods.
- No GC pauses on the path that decides "is this packet in scope".
- IronClaw's WASM-sandbox + gateway design works in Rust; we extend it.
- Cost: Rust is a smaller talent pool than Python. The trust path is small enough (low thousands of lines, not tens of thousands) that this is manageable.

**TypeScript for UI.**

- React/Next.js is the path of least resistance for web UIs. We have no reason to be exotic here.
- Type-safe IPC with the Python control plane via OpenAPI or tRPC over a generated client.

**WASM for skill-tier sandbox.**

- Skills generated by Phoenix or contributed by less-trusted signers should not run unrestricted on the worker.
- WASM has mature capability-based sandboxing, fast cold-start (sub-millisecond), and a small attack surface compared to a container.
- Promotion path: WASM canary → MicroVM standard execution. Skills earn trust over time.
- Cost: not every Python skill compiles to WASM trivially (`pyodide` exists, but with limits). We accept that high-privilege skills run in MicroVMs, not WASM; WASM is the sandbox for the *new* and *untrusted* tier.

## Consequences

- Multi-language build complexity: a single `make`/`just`/`mise` workflow.
- Cross-language IPC: gRPC or Cap'n Proto between Python control and Rust gateway. The trust boundary is the IPC schema.
- Documentation must clearly state "this lives in the trust path; PRs require security-review" for the Rust components.

## Revisit if

- WASM ecosystem stalls and skill-tier sandbox throughput becomes a bottleneck.
- The Rust components consistently underperform a Go equivalent in terms of contribution rate.
- Frontier-model Python SDKs are deprecated in favor of native-protocol clients in other languages.
